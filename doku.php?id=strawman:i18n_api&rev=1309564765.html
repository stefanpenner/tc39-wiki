EcmaScript i18n API

Introduction

There are a number of areas where current EcmaScript I18N APIs do not meet the needs of I18N sufficiently. Our goal is to develop better I18N ECMAScript APIs for both browser-hosted or standalone implementations.

These are proposed APIs and we welcome feedback (either inside of the document or on the mailing list).

Background

EcmaScript has limited support for i18n. The current API offers rudimentary sorting and formatting functions like Date.toLocaleString(). The API is also tied to the fixed, browser locale.

String sorting implementation varies among EcmaScript implementations used in browsers (Opera, Safari, Chrome, IE...), so it’s mostly useless, with unpredictable results (see Chromium bug 28604 and related V8 discussion).

Developers came up with two ways of solving some of these problems:

  • Implemented dedicated i18n libraries for formatting dates and currency and bundle it with a single-locale data.
  • Do all the i18n processing on the server and send down the results.

Both of these approaches have problems:

  • Server side processing introduces latency and requires on-line access and a server that would do i18n work. For instance, browser extensions or browser UIs written in EcmaScript do not have any server to talk to.
  • Dedicated libraries have various limitations, like size of the library and inconsistent API. The biggest problem is the size and complexity of data needed for collation process, which means that clients have to either use preprocessed data for sorting or use server to sort data. Moreover, multiple browser extensions may end up installing multiple copies of an I18N libraries or multiple I18N libraries with mostly overlapping data/APIs.

Goal

A consistent EcmaScript API would solve all of the problems:

  • Consistent, rich i18n API, similar to what modern OS/I18N libraries have today for C++ and Java.
  • Consistent i18n results (not 100% identical but comparable to the data collected in CLDR or locale support on major platforms)
  • Support for multiple locales per application. Locale identifiers would follow the Unicode standard (a superset of BCP47).
  • Reuse of objects that are more expensive to create, like collators.
  • No latency since there would be no calls made to a server.
  • No size increase for pages or extensions, since i18n support would be baked in.
  • Off-line processing.

Proposed API

LocaleInfo

LocaleInfo class is a global object and acts as an aggregator for other classes. This approach avoids pollution of global namespace and makes it easy to pass locale data as a whole around.

Constructor

// Constructs new LocaleInfo object with default languageID (implementation specific).
// New object has locale.options = {'languageID': defaultLanguageID, regionID: ''}
var locale = new LocaleInfo();
 
// Constructs new LocaleInfo object with languageID passed as a string.
// New object has locale.options = {'languageID': resolvedLanguageID, 'regionID': resolvedRegionID}
var locale = new LocaleInfo(languageID);
 
// Constructs new LocaleInfo object with settings passed in.
// All settings are optional.
// Valid keys are languageID (string or a list of strings) and regionID (string).
// New object has locale.options = {'languageID': resolvedLanguageID, 'regionID': resolvedRegionID}
var locale = new LocaleInfo(settings);

Parameters

  • languageID - ASCII string or a priority list of ASCII strings. Each string is in Unicode LDML format.
    • If no valid language id is provided, use default locale (implementation specific). Our recommendation for browser environment is to use navigator.language as default locale.
  • regionID - ASCII string, formatted as LDML region subtag. If region code is unknown or invalid it should be set to ZZ.
  • settings object (can be empty):
// languageID as string.
{ languageID: 'sr',
  regionID: 'RS' }
 
// languageID as priority list.
{ languageID: ['sr', 'de-AT-u-co-phonebk', 'fr-FR'] }
Priority list details
  • Priority list is a JavaScript array.
    • Each element is a string which contains one valid Unicode LDML language id.
    • Position in the list is only used to break ties.
    • Invalid language ids or non-string elements should be ignored.
  • Matching algorithm
    • Match base language only (but preserve extension in the final result).
    • Full match with supported language id should be picked if available.

Matching algorithm example: 

requested [A, B], supported {A, B}, pick A, order breaks tie.
requested [A, B], supported {A', B}, pick B. A' is a near perfect match.
requested [A, B], supported {A', B'}, implementation dependent.

LocaleInfo.NumberFormat

Constructor

Constructor is not a part of public API. Users should invoke LocaleInfo.prototype.numberFormat instead.

Parameters

  • currencyCode
    • 3-letter ISO 4217 currency code
    • Overrides currency code inferred from region ID.
  • style
    • decimal
      • max 3 fraction digits, min 0.
      • min significant digits 1 (not integer digits).
      • Grouping separator is present if language uses it.
      • negative sign placement is locale sensitive.
    • currency
      • locale dependent number of digits.
      • currency symbol placement is locale sensitive.
    • percent - max 0 fraction digits.
    • scientific is optional.
  • pattern - see Patterns section under LDML number format.
  • skeleton
    • see skeleton design doc at ICU.
    • Unlike ICU skeleton proposal, we don’t need !! mark to diferenciate skeleton from pattern.
// Decimal style - 1234.5
{style: 'decimal'}
 
// Currency style - $1234.5
{style: 'currency'}
 
// Percent style - 45%
{style: 'percent'}
 
// Pattern - given '@@##' and 1.23004 format produces 1.23
{pattern: '@@##'}

Methods

/**
 * Formats a number value.
 *
 * {number} value
 */
LocaleInfo.__NumberFormat.prototype.format(value);
 
/**
 * Creates a new LocaleInfo.NumberFormat object from the original using settings overrides.
 */
LocaleInfo.__NumberFormat.prototype.derive(settings);
Format error handling

If value passed to the format method is not a number type, or has NaN value, format method returns “NaN” string.

LocaleInfo.DateTimeFormat

Constructor

Constructor is not a part of public API. Users should invoke LocaleInfo.prototype.numberFormat instead.

Parameters

  • dateStyle
    • short
    • long
    • Support for medium and full styles is optional.
  • timeStyle
    • short
    • medium
    • Support for long and full styles is optional.
  • skeleton
    • A “skeleton” describing fields desired in the output date style.
    • yMd – Same as short dateStyle.
    • yyyyMMMMdEEEE – Same as long dateStyle, include day of week.
    • yyyyMMMMd – Long date without day of week, era only if necessary.
    • yMMM – Year/month pattern.
    • MMMd – Month/day pattern.
    • Support for other skeleton patterns is optional.
    • See LDML date format for full description of skeleton fields.
Calendar support

Use -u-ca- in the languageID to specify the calendar to use.

Parameter presedence and construction

If “skeleton” and any of “timeStyle”/”dateStyle” are passed in, we take “skeleton” value and ignore any of the styles.

Both “timeStyle” and “dateStyle” can be passed in at the same time.

If no settings are passed in, or if they are invalid, we assume dateStyle: ‘short’, timeStyle: ‘short’.

Symbol width

Date and time symbol methods accept width parameter that can be “abbreviated” or “wide”. Support for “narrow” is optional.

  • English locale would produce “J” for “narrow”, “Jan” for “abbreviated” and “January” for “wide”.

Examples

LocaleInfo examples

// Create locale info object for Serbian language, written in Cyrillic script.
// Set region to Serbia.
var locale = new LocaleInfo({languageID: 'sr-Cyrl-RS',
                             regionID: 'RS'});
 
// Create new locale that has the same properties as the previous, except for the region
// which we set to US.
var localeUSD = locale.derive({regionID: 'US'});

Collator examples

var locale = new LocaleInfo('en');
 
// Create a collator with default parameters.
var collator = locale.collator();
 
var data = ['some', 'random', 'strings'];
 
// Sort data using collator.
// Note that you can't directly pass collator.compare to sort(). 
data.sort(function(a, b) {return collator.compare(a,b);});
 
// Short version using bind()
data.sort(collator.compare(a, b).bind(collator));
 
// Assuming filenames is an array of object with type and name properties.
var compareFileName = function(a, b) {
    if (a.type == b.type) { return coll.compare(a.name, b.name);}
    if (a.type =="directory") return -1;
    return 1;
}
 
filenames.sort(compareFileName);

NumberFormat examples

var locale = new LocaleInfo('en');
 
var numberFormatter = locale.numberFormat({'style': 'decimal'});
numberFormatter.format(12.30);  // returns 12.3
 
var deLocale = new LocaleInfo({languageID: 'de', regionID: 'US'});
 
// Currency symbol gets inferred from the regionID (US -> USD).
nf = deLocale.numberFormat({'style': 'currency'});
 
// returns $12,30 - number is formatted according to de locale, but is using US currency symbol.
nf.format(12.3);
 
// Get EUR back.
nfe = nf.derive({currencyCode: "EUR"});
// returns €12,30 - number is formatted according to de locale, but is using EUR currency symbol.
nfe.format(12.3);
 
// Skeleton example
var koLocale = new LocaleInfo('ko');
var koCurrencyFmt = koLocale.numberFormat({skeleton: '¤##,000.00', currencyCode: 'EUR'});
// Returns €98,745,612.34
koCurrencyFmt.format(98745612.34);
 
koCurrencyFmt = koCurrencyFmt.derive({skeleton: '¤¤##,000.00'});
// Returns EUR98,745,612.34
koCurrencyFmt.format(98745612.34);
 
koCurrencyFmt = koCurrencyFmt.derive({skeleton: '¤¤¤##,000.00'});
// Returns 98,745,612.34 유로화
koCurrencyFmt.format(98745612.34);

DateTimeFormat examples

var locale = new LocaleInfo('en');
 
var df = locale.dateTimeFormat({dateStyle: 'short'});
 
// Returns current date formatted as 30.6.11
df.format();
 
var deLocale = new LocaleInfo('de');
df = deLocale.dateTimeFormat({dateStyle: 'medium', timeStyle: 'medium'});
// Returns current time/date formatted as 30.06.2011 16:46:14.
df.format();
 
var df2 = locale.dateTimeFormat({skeleton: 'yMMM'});
// Returns “Jun 2011".
df2.format();
 
// Get symbols in Serbian.
var srLocale = new LocaleInfo('sr');
var dts = locale.dateTimeFormat();
 
// width defaults to wide.
var months = dts.getMonths();  // Retuns ['Januar',..., 'Decembar']
